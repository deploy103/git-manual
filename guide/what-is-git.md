# What is Git

`Git` 은  
파일의 변경 이력을 기록하고 관리하는 버전 관리 도구이다.

쉽게 말하면  
코드를 작성하면서  
**누가, 언제, 무엇을, 어떻게 바꿨는지 기록해 두는 시스템**이라고 보면 된다.

---

## 1. 왜 Git 을 사용하는가

프로젝트를 진행하다 보면  
파일이 계속 바뀌게 된다.

예를 들어 아래와 같은 상황이 자주 생긴다.

- 이전 버전으로 되돌아가고 싶음
- 어떤 파일을 누가 수정했는지 보고 싶음
- 실수로 파일을 망가뜨림
- 여러 사람이 같은 프로젝트를 같이 작업함
- 기능별로 작업을 나눠서 관리하고 싶음

이럴 때 Git 이 있으면  
변경 내용을 기록하고,  
필요할 때 이전 상태로 돌아가고,  
협업도 체계적으로 할 수 있다.

---

## 2. Git 이 하는 일

Git 은 프로젝트의 파일 상태를 계속 추적한다.

그리고 특정 시점마다  
"지금 상태를 하나의 기록으로 남기자" 라고 저장할 수 있는데  
이 기록이 바로 `commit` 이다.

즉, Git 은  
프로젝트의 변경 과정을 commit 단위로 쌓아 가는 도구이다.

---

## 3. Git 을 안 쓰면 생기는 문제

Git 이 없으면  
보통 파일 이름을 이런 식으로 관리하게 된다.

- final.c
- final_real.c
- final_real_last.c
- final_real_last_fix.c

이렇게 되면  
어느 파일이 진짜 최신인지 헷갈리고,  
이전 상태로 돌아가기도 어렵다.

반면 Git 을 쓰면  
파일 이름을 이상하게 늘리지 않고도  
변경 이력을 전부 기록할 수 있다.

---

## 4. Git 의 핵심 개념

Git 을 처음 볼 때는  
아래 4가지만 먼저 이해하면 된다.

- `repository` : Git 이력을 관리하는 저장소
- `commit` : 특정 시점의 기록
- `branch` : 작업 흐름을 나누는 가지
- `merge` : 나뉜 작업을 다시 합치는 것

---

## 5. repository 란

`repository` 는  
Git 이 프로젝트의 변경 이력을 저장하는 공간이다.

보통 프로젝트 폴더 안에  
`.git` 이라는 숨김 폴더가 생기는데  
여기에 Git 정보가 저장된다.

즉, 어떤 폴더에서 `git init` 을 하면  
그 폴더는 Git 저장소가 된다.

---

## 6. commit 이란

`commit` 은  
"현재 프로젝트 상태를 하나의 기록으로 저장한 것" 이다.

예를 들어

- 로그인 기능 추가
- 업로드 오류 수정
- README 문서 정리

같은 작업을 끝낸 뒤  
그 상태를 commit 으로 남길 수 있다.

이 commit 들이 쌓이면서  
프로젝트의 작업 이력이 만들어진다.

---

## 7. branch 란

`branch` 는  
작업 흐름을 나누는 독립적인 가지라고 보면 된다.

예를 들어

- `main` : 안정적인 기본 버전
- `feat/login` : 로그인 기능 작업
- `fix/upload-error` : 업로드 오류 수정

이런 식으로 나눠서 작업할 수 있다.

즉, branch 를 쓰면  
메인 작업을 망가뜨리지 않고  
새 기능이나 수정 작업을 따로 진행할 수 있다.

---

## 8. merge 란

`merge` 는  
따로 작업하던 branch 내용을  
다시 하나로 합치는 작업이다.

예를 들어  
`feat/login` 에서 로그인 기능을 만든 뒤  
작업이 끝나면 `main` 에 합친다.

이 과정을 merge 라고 한다.

---

## 9. Git 과 GitHub 는 다르다

많이 헷갈리지만  
`Git` 과 `GitHub` 는 같은 것이 아니다.

### Git
내 컴퓨터에서 버전 관리를 해 주는 도구

### GitHub
Git 저장소를 인터넷에 올려서  
백업, 공유, 협업할 수 있게 해 주는 서비스

즉, Git 이 본체이고  
GitHub 는 Git 저장소를 올려 두는 공간이라고 보면 된다.

---

## 10. 한 줄 예시 흐름

아래처럼 생각하면 된다.

1. 파일 수정  
2. Git 에 변경 사항 추가  
3. commit 으로 기록  
4. 필요하면 GitHub 에 push  
5. 다른 사람과 공유하거나 협업  

---

## 11. Git 이 특히 좋은 상황

Git 은 아래 상황에서 특히 강하다.

- 프로젝트를 오래 관리할 때
- 실수했을 때 되돌리고 싶을 때
- 여러 기능을 따로 개발할 때
- 협업할 때
- 작업 기록을 남기고 싶을 때

---

## 12. 한 줄 정리

`Git` 은  
**프로젝트 파일의 변경 이력을 기록하고, 되돌리고, 나누고, 합칠 수 있게 해 주는 버전 관리 도구**이다.
